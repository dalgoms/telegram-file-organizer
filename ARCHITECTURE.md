# Telegram File Organizer Bot — 아키텍처 & 설계 문서

> 대화 기반 설계 과정 기록 | 2026-03-19

---

## 1. 프로젝트 개요

텔레그램에서 폴더 경로를 보내면 AI가 파일을 분석하여 자동으로 정리해주는 봇.
"나 대신 커서가 해주던 파일 정리를, 커서 없이도 쓸 수 있게" 가 출발점.

### 핵심 가치
- **기록의 허들을 0으로** — 텔레그램 한 줄이면 정리 시작
- **AI가 대신 판단** — 어떤 폴더에 넣을지 사람이 안 정해도 됨
- **실수해도 복구** — 모든 이동 기록을 남기고 Undo 가능
- **사용자가 결정** — 자동 실행 없음, 모든 액션은 사용자 버튼 클릭 필요

---

## 2. 설계 동기 — 대화에서 나온 핵심 질문들

### Q1. "커서가 해주던 파일 정리를 봇으로 만들 수 있어?"
→ 가능. 텔레그램 봇 + Python pathlib/shutil + OpenAI API 조합.

### Q2. "장기간 근로한 직원들이 진짜 편하다고 느끼려면?"
→ 단순 정리만으로는 부족. 수년간 쌓인 파일의 고통을 해결해야 함:
- 같은 파일이 여러 폴더에 중복
- "보고서_최종_수정_진짜최종" 버전 지옥
- 용량 먹는 파일이 어디인지 모름
- 5년 치 파일이 한 폴더에 섞여 있음

→ `/dup`, `/ver`, `/size`, `/old`, `/report` 분석 기능 추가

### Q3. "집 컴퓨터랑 회사 컴퓨터 구분해서 명령할 수 있어?"
→ 기기별 봇 인스턴스 + `DEVICE_NAME` 환경변수로 해결.
  모든 응답에 `[집PC]`, `[회사PC]` 태그 표시.

### Q4. "정리할 때 언제 정리했는지 알 수 있으면 좋겠어"
→ 정리 실행 시 `정리_YYYY-MM-DD/` 상위 폴더를 자동 생성.
  여러 번 정리해도 날짜별로 깔끔하게 분리.

### Q5. "AI가 멋대로 삭제하면 어떡해?"
→ 이 봇은 사용자가 명시적으로 버튼을 누르지 않는 한 아무것도 하지 않음.
  삭제는 2단계 확인을 거침. 자동 삭제/자동 정리 기능은 의도적으로 배제.

## 3. 시스템 아키텍처

```
텔레그램 (사용자)
│
│  /scan D:\Downloads
│
▼
bot.py ─── 명령어 라우팅 + 인라인 버튼 처리
│
├── scanner.py ─── 파일 시스템 스캔
│   └── FileInfo, ScanResult 데이터 클래스
│
├── classifier.py ─── GPT-4o-mini로 AI 분류
│   └── 파일 목록 → JSON 카테고리 매핑
│
├── organizer.py ─── 실제 파일 이동 + Undo
│   └── 정리_YYYY-MM-DD/ 날짜 폴더 자동 생성
│   └── history.json에 모든 이동 기록
│
├── analyzer.py ─── 장기 근속자용 분석 도구
│   ├── find_duplicates() ─── MD5 해시 기반 중복 탐지
│   ├── find_version_chains() ─── 최종/수정/v1 패턴 인식
│   ├── analyze_size() ─── 용량 TOP + 확장자별 통계
│   ├── suggest_archive() ─── 연도별 아카이브 제안
│   └── search_files() ─── 키워드 검색
│
├── rules.py ─── 커스텀 분류 규칙
│   └── rules.json에 패턴 → 폴더 매핑 저장
│
├── shortcuts.py ─── 경로 바로가기 (/path)
│   └── paths.json에 별칭 → 경로 매핑 저장
│
├── stats.py ─── 정리 통계 + 업적 시스템 (/stats)
│   └── stats.json에 누적 기록 저장
│
├── scheduler.py ─── 예약 스캔 (/schedule)
│   └── schedules.json에 예약 목록 저장
│
├── cloud/gdrive.py ─── Google Drive API 연동
│
└── config.py ─── 환경변수 + 설정 관리
    └── DEVICE_NAME으로 기기 식별
    └── QUICK_PATHS로 사용자 폴더 자동 감지
```

## 4. 데이터 흐름

### 정리 흐름 (메인)

```
사용자 입력          처리 단계                     결과
──────────     ──────────────────     ──────────────
/scan 경로  →  scanner.scan_local()   →  ScanResult (파일 목록)
            →  classifier.classify()  →  {folders: {카테고리: [파일]}}
            →  rules.apply_rules()    →  커스텀 규칙 우선 적용
            →  미리보기 전송           →  정리_YYYY-MM-DD/ 구조 표시
                                      →  [정리하기] [삭제하기] [그만두기] 버튼

정리 실행   →  organizer.execute()    →  정리_YYYY-MM-DD/ 폴더 생성 + 파일 이동
            →  history.json 저장      →  Undo 가능 상태

/undo       →  organizer.undo_last()  →  원위치 복원 + 날짜 폴더 삭제
```

### 분석 흐름 (장기 근속자용)

```
/dup    →  크기 비교 → MD5 해시 비교 → 중복 그룹 리포트  → [정리/삭제] 버튼
/ver    →  정규식으로 버전 키워드 제거 → 버전 체인 그룹핑  → [정리/삭제] 버튼
/size   →  크기 정렬 TOP N + 확장자별 집계               → (통계만)
/old    →  수정 날짜 기준 연도별 분류 → 아카이브 제안     → [정리/삭제] 버튼
/find   →  마지막 스캔 결과에서 파일명 매칭              → [정리/삭제] 버튼
/report →  위 전부를 순차 실행 → 종합 진단               → [정리/삭제] 버튼
```

### 삭제 흐름 (2단계 확인)

```
[삭제하기] 버튼  →  파일 목록 + 총 용량 표시  →  [정말 N개 삭제할게요] / [취소]
                                                       │
                                              확인 시 → 파일 삭제 실행
                                              취소 시 → "파일은 그대로예요"
```

## 5. 핵심 설계 결정

### 5.1 "사용자가 결정" 원칙

```
스캔 → AI 분류 → [미리보기] → 사용자 확인 → 실행
                     ↑
               이 단계가 핵심
```

이 봇은 **자동 실행 기능이 의도적으로 없습니다.**
- 자동 정리 없음
- 자동 삭제 없음
- 예약 스캔도 알림만 보냄 (실행은 사용자가 결정)

**이유**: 파일 관리 도구의 첫 번째 원칙은 "예측 가능해야 한다."
사용자가 모르는 사이에 파일이 이동되거나 삭제되면 신뢰를 잃는다.

### 5.2 날짜 폴더 — 정리 이력의 시각화

```
Downloads/
  ├─ 정리_2026-03-15/    ← 3/15에 정리한 파일
  │   ├─ 문서/보고서/
  │   └─ 이미지/
  └─ 정리_2026-03-19/    ← 3/19에 정리한 파일
      ├─ 문서/기획/
      └─ 데이터/
```

**이유**: "이 파일 언제 정리했지?" 를 별도 이력 조회 없이 폴더명만으로 확인 가능.
여러 번 정리해도 날짜별로 분리되어 섞이지 않음.

### 5.3 중복 탐지 — 2단계 필터링

```
전체 파일 → [1단계: 크기 비교] → 크기 같은 파일만 → [2단계: MD5 해시] → 진짜 중복
```

**이유**: 해시 계산은 디스크 I/O가 큼. 크기가 다르면 중복이 아니므로 먼저 걸러냄.
대용량 파일은 앞뒤 8KB만 해시하여 속도 최적화.

### 5.4 버전 체인 — 한국어 + 영어 패턴

인식하는 버전 패턴:
| 패턴 | 예시 |
|------|------|
| 한글 버전 | 최종, 수정, 진짜최종, 최최종, 초안 |
| 영문 버전 | final, revised, rev, draft, copy |
| 숫자 버전 | v1, v2, v3 |
| 날짜 버전 | 20260318 (8자리 숫자) |
| 괄호 버전 | (1), (2), (3) |

**방식**: 정규식으로 버전 표시를 제거 → 나머지가 같으면 같은 파일의 버전으로 판단

### 5.5 기기 식별 — 환경변수 방식

```
# 집 PC
DEVICE_NAME=집PC
TELEGRAM_BOT_TOKEN=집봇_토큰

# 회사 PC  
DEVICE_NAME=회사PC
TELEGRAM_BOT_TOKEN=회사봇_토큰
```

**이유**: 가장 단순한 방식. 서버/DB/네트워크 없이 각 PC가 독립 실행.
한 봇 토큰은 한 인스턴스만 가능하므로 (Telegram Polling 제약) 봇을 2개 만든다.

### 5.6 안전장치 체계 (3단계 방어)

**실행 전 — 차단과 검증**
| 장치 | 구현 |
|------|------|
| 미리보기 필수 | 정리 전 항상 분류안 표시 + 확인 버튼 |
| 시스템 폴더 차단 | BLOCKED_PATHS로 C:\Windows, C:\Program Files 등 원천 차단 |
| 경로 탈출 방어 | `_sanitize_folder_name()`에서 `../`, `.` 패턴 자동 제거 |
| 루트 외부 파일 차단 | `src.resolve()`가 root 내부인지 검증 |
| 스캔 만료 | 5분 경과 시 자동 만료 — 변경된 파일로 정리 방지 |
| 버튼 세션 검증 | 이전 스캔 버튼 클릭 시 무효 처리 |
| 스캔 제한 | MAX_FILES_PER_SCAN = 500 (메모리/API 비용 보호) |

**실행 중 — 실시간 보호**
| 장치 | 구현 |
|------|------|
| 실시간 이력 저장 | 매 파일 이동마다 history.json 즉시 저장 |
| 원자적 쓰기 | `os.replace()`로 JSON 손상 방지 (크래시/정전 대응) |
| 심볼릭 링크 건너뜀 | 외부 폴더로의 의도치 않은 파일 이동 방지 |
| 파일 잠금 감지 | PermissionError 시 "사용 중인 파일" 안내 |
| 경로 길이 체크 | Windows 260자 제한 감지 |
| 파일명 충돌 | 동일명 존재 시 `파일_1.ext` 자동 번호 부여 |
| 시스템 파일 보호 | SKIP_DIRS, SKIP_FILES로 .git, node_modules 등 자동 제외 |

**실행 후 — 복구**
| 장치 | 구현 |
|------|------|
| Undo | history.json에 from/to 경로 기록, 역순 복원 |
| 정밀 폴더 삭제 | undo 시 봇이 만든 폴더만 삭제 (원래 빈 폴더 보존) |
| 이력 제한 | 최근 50건만 보관 (무한 증가 방지) |

**삭제 안전장치 — 2단계 확인**
| 단계 | 동작 |
|------|------|
| 1차 확인 | 삭제 대상 파일 목록 + 총 용량 표시 |
| 2차 확인 | "정말 N개 삭제할게요" 재확인 버튼 |
| 취소 | 어느 단계에서든 취소 가능 |

## 6. 모듈 상세

### bot.py — 텔레그램 핸들러 (약 900줄)
- 20+개 명령어 핸들러 등록
- `PENDING_SCANS`: 스캔 결과 임시 저장 (확인 대기)
- `LAST_SCANS`: 마지막 스캔 캐시 (분석 명령어 재사용)
- `FIND_RESULTS`: 분석 결과 파일 캐시 (정리/삭제 액션용)
- `TAG`: 모든 응답에 `[기기명]` 접두사 부착
- `_show_action_buttons()`: 분석 결과 후 정리/삭제 버튼 공통 함수
- `_scheduled_scan_job()`: 매 분 예약 스캔 체크
- 일반 메시지도 경로 패턴 감지 시 자동 스캔

### scanner.py — 파일 스캔 (약 120줄)
- `FileInfo` 데이터 클래스: name, path, extension, size, modified
- `ScanResult`: 파일 목록 + 에러 + 건너뛴 항목 수
- 재귀/비재귀 스캔 지원 (`-r` 옵션)
- 확장자별 현황 요약

### classifier.py — AI 분류 (약 150줄)
- GPT-4o-mini 사용 (빠르고 저렴, 분류에 충분)
- temperature 0.2 (일관성 우선)
- max_tokens 4000 (긴 파일 목록 대응)
- 파일 200개 제한 (안정성)
- `_repair_json()`: 잘린 GPT 응답 자동 복구
- 미리보기에 `정리_YYYY-MM-DD/` 날짜 폴더 표시

### organizer.py — 파일 이동 + Undo (약 250줄)
- `execute_organization()`: `정리_YYYY-MM-DD/` 폴더 생성 → 파일 이동 → 이력 저장
- `undo_last()`: 마지막 이력 역순 복원 → 날짜 폴더 포함 빈 폴더 삭제
- `history.json`에 세션 단위로 기록
- 매 파일 이동마다 원자적 이력 저장

### analyzer.py — 분석 도구 (약 280줄)
- `find_duplicates()`: 크기→해시 2단계 중복 탐지
- `find_version_chains()`: 11개 정규식 패턴으로 버전 키워드 제거 후 그룹핑
- `analyze_size()`: TOP N + 확장자별 용량/개수 통계
- `suggest_archive()`: 올해가 아닌 파일을 연도별 분류
- `search_files()`: 키워드 포함 파일 검색

### shortcuts.py — 경로 바로가기 (약 80줄)
- "다운로드", "바탕화면" 등 한글 별칭 지원
- 사용자 정의 별칭 추가/삭제
- 내장 경로는 `os.expanduser("~")`로 자동 감지

### stats.py — 정리 통계 + 업적 (약 120줄)
- 총 정리 파일 수, 세션 수, 연속 정리일 추적
- 칭호 시스템: "정리 새싹" → "정리 장인" → "폴더 마스터"
- 업적: "첫 정리 완료", "100개 파일 돌파", "3일 연속" 등

### scheduler.py — 예약 스캔 (약 80줄)
- 요일 + 시간 기반 예약 (예: "월 09:00")
- 예약 시간에 자동 스캔 → 알림만 발송 (실행은 사용자 결정)
- schedules.json에 영구 저장

### rules.py — 커스텀 규칙 (약 90줄)
- glob 패턴 기반 매칭 (`*.pptx` → `문서/프레젠테이션`)
- AI 분류 결과에 커스텀 규칙을 우선 적용
- rules.json에 영구 저장

### cloud/gdrive.py — Google Drive (약 150줄)
- OAuth 2.0 인증 (credentials.json → token.json)
- 폴더 스캔, 파일 이동, 폴더 생성
- 로컬과 동일한 ScanResult 형식으로 통일

### config.py — 설정 (약 50줄)
- 환경변수 로드 (`.env` 절대경로 명시)
- DEVICE_NAME 자동 감지 (미설정 시 hostname 사용)
- BLOCKED_PATHS: 시스템 폴더 차단 목록
- QUICK_PATHS: 사용자 폴더 자동 감지 (바탕화면, 다운로드, 문서, 사진, 음악, 동영상)
- SKIP_DIRS/SKIP_FILES 목록

## 7. 기술 스택

| 계층 | 기술 | 선택 이유 |
|------|------|-----------|
| 인터페이스 | Telegram Bot API | 매일 여는 앱, 허들 최저 |
| 봇 프레임워크 | python-telegram-bot[job-queue] 21.10 | 비동기 지원, 버튼/콜백 내장, 예약 작업 |
| AI 분류 | OpenAI gpt-4o-mini | 파일 분류는 단순 작업, 빠르고 저렴 |
| 파일 조작 | pathlib + shutil | Python 표준 라이브러리, 안정적 |
| 중복 탐지 | hashlib MD5 | 빠른 해시, 보안이 아닌 비교 목적 |
| 클라우드 | Google Drive API v3 | 한국 직장인 점유율 높음 |
| 설정 | python-dotenv | .env 파일 기반, 기기별 설정 분리 |
| 데이터 저장 | JSON 파일 (원자적 쓰기) | DB 불필요, 이력/규칙/통계는 JSON으로 충분 |

## 11. UX 설계 — 토스 원칙 적용

### 적용한 핵심 원칙

| 원칙 | 적용 |
|------|------|
| 한 화면에 한 행동 | `/start`에서 20개 명령어 나열 대신, "바로 시작하기" 하나에 집중 |
| 쉬운 말 | "분류안대로 정리 실행" → "5개 폴더로 정리하기" |
| 두려움 제거 | 모든 실행 버튼 옆에 "잘못되면 /undo 한 번이면 돌아와요" 안내 |
| 상태 항상 표시 | "스캔 중" → "'Downloads' 폴더를 살펴보고 있어요..." |
| 동적 경로 | 하드코딩 경로 대신 `os.expanduser("~")`로 실제 PC 경로 자동 감지 |
| 결과에서 바로 액션 | 분석 후 "정리하기" / "삭제하기" 버튼으로 즉시 행동 가능 |

### 메시지 톤 개선

| Before (개발자 관점) | After (사용자 관점) |
|---------------------|---------------------|
| `[정리 완료] 12개 파일 이동` | `깔끔하게 정리했어요! (12개 파일)` |
| `[되돌리기 완료] 12개 파일 원위치` | `원래대로 돌려놨어요. (12개 파일)` |
| `AI 분류 분석 중...` | `파일 47개를 AI가 분류하고 있어요...` |
| `정리를 취소했습니다.` | `취소했어요. 파일은 그대로예요.` |
| `정리 실행` 버튼 | `5개 폴더로 정리하기` 버튼 |
| `대기 중인 스캔이 없습니다.` | `스캔 결과가 만료되었어요.` |

### 점진적 공개 (Progressive Disclosure)

```
/start  →  "경로 하나만 보내보세요" (1가지 행동)
           ↓ 성공 경험
/help   →  1. 정리 → 2. 진단 → 3. 규칙 → 4. 드라이브 → 5. 편의 (여정 순서)
           ↓ 익숙해지면
/report →  종합 진단으로 고급 기능 자연 노출
           → 결과에서 바로 정리/삭제 버튼
```

## 12. 설계 원칙

| 원칙 | 적용 |
|------|------|
| **사용자가 결정** | 자동 실행 없음. 모든 액션은 사용자 버튼 클릭 필요. 삭제는 2단계 확인 |
| **확인 후 실행** | 파일 이동 전 반드시 미리보기. 자동화의 신뢰는 예측 가능성에서 온다 |
| **실패 안전** | 모든 이동을 기록하고 Undo 가능. 시스템 파일은 자동 스킵 |
| **점진적 복잡도** | 경로만 보내면 동작 → 분석 명령어 → 커스텀 규칙 → 드라이브 |
| **기기 독립** | 서버 없이 각 PC에서 독립 실행. .env만 다르면 됨 |
| **비용 효율** | gpt-4o-mini로 충분. DB 대신 JSON. 인프라 비용 0 |
| **한국어 우선** | 폴더명, 버전 패턴, 메시지 모두 한국어 기반 설계 |
| **날짜 기록** | 정리 시 날짜 폴더 자동 생성 — 이력의 시각적 관리 |
